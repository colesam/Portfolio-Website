Description

This project is a web application I built with my partner, Belinda Huang, for our CISC 375 Web Development course at the University of St. Thomas.

The project was built using HTML5, CSS3, Bootstrap, NodeJS, Express, HoganJS, and SQLite 3. It uses a SQLite database our instructor created by combining the databases found on imdb's website. The combined database is available for download here.

It should be noted that there were no styling requirements for this project. In order to focus our efforts on the back end, we used the bootstrap grid system to handle our layout, and did little custom CSS.

Features

  • Allows users to search for both Titles and People
  • Features a page to view information about a single title or person
  • Asynchronously loads data on the page to increase page load performance
  • Titles and people are editable pages and it will save your changes to the database
  • Features about the project page with short bios of each partner, as well as a video demonstration of the site

Code I Wrote

My partner and I both contributed a lot to this project. We also decided that it would be most efficient if we each focused on separate parts of the project. This strategy still required a lot of teamwork and communication between us, while allowing us to take advantages of our individual strengths.

Below you will find an explanation of what parts of the project I focused on, and where to find those parts within the project directory.

FOLDER STRUCTURE

I generated the structure of the project using the express-generator module from NPM. This set up the structure of the project with express and hoganJS already integrated.

VIEWS

I created all the view templates found in the root/views directory. The exception to this are the people and people-edit templates which were written by my partner, Belinda Huang.

Templating was necessary for displaying information on titles and people easily. We simply needed to create a template and then supply that template with information from the SQL database. Templates also allowed us to create partial files for the header and footer. This kept the actual templates clear of copy and pasted html.

I chose to use the HoganJS templating engine for my views because it was very easy to set up with the express-generator. It turned out that it was missing some key features that caused difficulties (see below). If I were to redo the project, I would likely choose a different templating engine.

ROUTES and DATA

You will find the routes for the project in the root/routes/index.js directory. This allowed us to have more control over where a URL directs a user. I wrote the routes that deal with titles and my partner wrote the routes that deal with people. All routes either retrieve or post data to the database. Methods for retrieving or updating data were organized into custom modules, found in the root/custom_modules directory.

I also wrote all of the logic for the AJAX routes. These are routes that JQuery code could access in order to load resources asynchronously. For example, instead of waiting for the title poster to load before sending the view, I sent the view immediately and then requested the poster from the /ajax/titles/posters/:title_id route using JQuery.

An interesting ajax route to look at is the /ajax/cast/:title_id route. This required me to write a separate function, data_retriever.getCastFromTitleId(). This was one of the more difficult parts of the project and is outlined in the difficulties section below.

Difficulties

GET CAST

The most difficult part of this project was writing the getCastFromTitleId() function in the get_data.js custom module. The purpose of this function is to take a title_id and find the cast for that title. Unfortunately, the way the database was set up makes this very difficult.

The database keeps the cast split into two tables, Principals and Crew. There Schemas looked like this:

Principals(tconst, ordering, nconst, category, job, characters)

Crew(tconst, directors, writers)

Principals was very easy to join with the Names database because it had an nconst column that acts as a name id. Crew on the other handle supplies you with a comma-delimited list of name ids for directors and writers, given a particular title id. To make matters worse, there were duplicates between Principals and Crew. For example, sometimes a director was included in Principals and Crew, sometimes only in Principals, and sometimes only in Crew.

I attacked this problem by writing a second helper function getDataByIdArray() inside get_data.js. I would convert the comma-delimited list of name ids into an array and pass it to this function. The function would return a Promise.all object that I could use to get an array of rows from the Name table.

By abstracting out the most complicated part of getting the Cast, I was able to clean up my getCastFromTitleId() to only do the following. First, use a promise to get an array of rows from the Name table that correspond to Principals from the title we are working with. Next, use a promise to get an array of rows from the Name table that correspond to Crew members. Once both of these promises resolve, combine the two arrays while using name ids to ensure there are no duplicates.

LOGIC-LESS TEMPLATING

About half way through the project I realized that there was a major problem with the HoganJS templating engine. The problem was that .hjs templates do not support very robust conditional checks. This made doing things like setting default values of form inputs very difficult.

An example of how I overcame this problem can be seen in the root/views/title-edit.hjs template. The .hjs templates only support whether or not a variable has been set, but do not support checking if the variable has been set to a particular value.

I wanted to have the genres associated with the title being edited to be pre-selected in the form. Within the /titles/:title_id/edit route, a video object is passed to the title-edit.hjs view. For each genre associated with this video, I added an attribute to the video object so that I could set the selected options with the following line within each option element of the select input:

{{#video.Romance}}selected{{/video.Romance}}

This would check if the passed video object had a 'Romance' attribute. If it did, it would set the romance option to selected, solving my logic-less templating issue. In the future I would choose a different templating engine to avoid having to hack this functionality together.
